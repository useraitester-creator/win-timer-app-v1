# Дизайн: виды задач и «план на день»

Дата: 2026-06-11
Статус: на ревью

## Цель

Перейти от модели «задача привязана к дню + авто-копии (продолжения)» к модели
**одной постоянной задачи** с тремя представлениями и ручным управлением планом на день:

1. **Все задачи** — все задачи, которые когда-либо были в приложении.
2. **План на сегодня** — что выбрано на сегодня; по умолчанию заполняется
   незавершёнными задачами из вчерашнего плана.
3. **В работе** — все незавершённые задачи.

Плюс: кнопка **добавить/убрать** задачу из плана на сегодня.

## Почему меняем

Сейчас `ensure_rollover()` каждый новый день создаёт **копию** незавершённой
задачи с суффиксом «(продолжение)» (`continuation_of` ссылается на исходную).
Одна задача за неделю превращается в 5 записей. Для вида «Все задачи» это даёт
дубли, а понятие «план» отсутствует. Переходим на одну сквозную задачу.

## Модель данных

`win_timer_app/models.py` — `Task`:

- **Добавить** `planned_days: list[str]` — ISO-даты дней, когда задача в плане.
  По умолчанию `[]`. Сериализуется в `to_dict`/`from_dict`.
- `day` — оставляем как «день создания» (для сортировки/отображения), но больше
  не используем для группировки и копирования.
- `continuation_of` — оставляем поле для совместимости со старыми данными, новые
  задачи его не используют.

Время:
- **«Сегодня»** у задачи = сумма длительностей сессий, у которых `started_at`
  приходится на сегодняшнюю дату.
- **«Всего»** = все сессии (`Task.total_seconds`, уже есть).
- Сессии не пересекают полночь: активная задача на стыке дней паузится в 23:59:59
  (существующий `_close_cross_day_active_task` сохраняем).

## Контроллер

Виды (возвращают `list[Task]`, сортировка — по `created_at` убыв., активная задача
выше):
- `tasks_all()`
- `tasks_in_progress()` — `status != COMPLETED`
- `tasks_today_plan()` — `today in planned_days`

План:
- `in_today_plan(task) -> bool`
- `add_to_plan(task_id)` — добавляет сегодняшний день в `planned_days`, `save()`
- `remove_from_plan(task_id)` — убирает сегодняшний день, `save()`

Перенос плана на новый день — `ensure_plan_rollover(today=None)`:
- Если в `ui["plan_rollover_day"]` уже стоит сегодня — ничего не делаем
  (чтобы убранную сегодня задачу не вернуть обратно).
- Иначе: для каждой задачи, где `status != COMPLETED` и `вчера in planned_days`,
  добавляем `today` в `planned_days`.
- Записываем `ui["plan_rollover_day"] = today`, `save()`.
- Вызывается на старте и при тике, как сейчас `ensure_rollover`.

Время за день:
- `today_seconds(task, today=None) -> int`
- `today_total_seconds(today=None) -> int` — сумма по всем задачам (для шапки).

Старый `ensure_rollover()` (копирование «продолжений») **удаляется**; его вызовы
заменяются на `ensure_plan_rollover()`. `_base_title` больше не нужен.

Параметр `today=None` у методов нужен для тестируемости (в тестах передаём
конкретную дату вместо `date.today()`).

## Миграция существующих данных

Выполняется один раз при загрузке, до `ensure_plan_rollover`, помечается флагом
`ui["schema_version"] = 2`.

Алгоритм:
1. Если `schema_version >= 2` — выходим.
2. **Свернуть цепочки «продолжений»:** сгруппировать задачи по корню (идём по
   `continuation_of` до задачи без него). Для каждой цепочки:
   - переносим все сессии членов цепочки в корневую задачу;
   - `planned_days(root)` = множество всех значений `day` в цепочке;
   - `status(root)` = статус последнего (по `day`) члена цепочки;
   - `title(root)` очищаем от суффикса «(продолжение)»;
   - остальные члены цепочки удаляем.
3. Для задач без цепочек: если `planned_days` пуст — выставить `[day]`.
4. `ui["plan_rollover_day"] = today` (чтобы перенос не сработал сразу после
   миграции).
5. `schema_version = 2`, `save()`.

Гарантия: ни одна сессия не теряется. Перед реализацией миграция тестируется на
копии реального `data.json` пользователя (read-only), проверяется суммарное время
до/после.

## UI (`main_window.py`)

Вкладка «Задачи»:
- Вместо чекбокса «Только незавершённые» — **сегментированный переключатель**
  из трёх кнопок: `[План на сегодня | В работе | Все задачи]`, активный
  подсвечен. По умолчанию «План на сегодня». Выбор сохраняется в
  `ui["tasks_view"]`.
- Под переключателем — строка «Сегодня всего: HH:MM:SS».
- **Плоский список** задач выбранного вида (без `DaySection`). Если пусто —
  подсказка («Нет задач в плане — добавь из «В работе» или «Все задачи»»).

`TaskRow`:
- Время: «Сегодня: HH:MM:SS · Всего: HH:MM:SS».
- Кнопка плана: «В план» (если сегодня нет в `planned_days`) / «Убрать из плана»
  (если есть).
- Существующее сохраняем: Старт/Стоп/Завершить, История, удалить, «Открыть в Б24».

Правая панель (текущая задача) — без изменений.

Поведение:
- Убрать из плана активную (запущенную) задачу можно — она продолжает идти и
  остаётся в «В работе».
- Завершённая задача уходит из «В работе», остаётся в «Все задачи»; в плане за
  сегодня показывается зачёркнутой (день из `planned_days` не убираем).

## Тестирование

TDD (юнит, без сети):
- `today_seconds` / `today_total_seconds` (сессии за сегодня vs другой день).
- Фильтры видов (`tasks_all`/`tasks_in_progress`/`tasks_today_plan`).
- `add_to_plan` / `remove_from_plan` (идемпотентность, персист).
- `ensure_plan_rollover`: перенос вчерашнего незавершённого плана на сегодня;
  не переносит завершённые; не возвращает убранную сегодня (повторный вызов —
  no-op).
- Миграция: цепочка «продолжений» → одна задача с объединёнными сессиями и
  корректными `planned_days`; суммарное время сохранено.

UI — проверка запуском (offscreen-драйвер + скриншоты): переключатель, плоский
список, кнопка «В план/Убрать», время в строке.

## Вне рамок (отдельные шаги)

- Шаг 2 интеграции: выгрузка времени в СПА 1092.
- Вкладка «Фокус» — без изменений.
